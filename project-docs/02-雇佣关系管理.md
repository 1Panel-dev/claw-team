# 需求 2：雇佣关系管理

## 1. 需求概述

雇佣关系是 1Claw 系统的核心关系模型，定义了人类（老板）与 OpenClaw（雇员）之间的归属关系。所有后续功能（对话、任务分配等）都基于此关系进行权限控制。

---

## 2. 系统架构

### 2.1 四大核心模块

1Claw 系统整体架构设计为四大核心模块：

| 模块 | 用户群体 | 架构形式 | 核心职责 |
|------|----------|----------|----------|
| 管理系统 | 系统管理员 | 前后端分离 | 账号管理、OpenClaw 管理、全局管控 |
| 人类 Web 客户端 | 人类（老板） | Web 应用 | 信息查看、通讯交互、任务分配 |
| OpenClaw 通道插件 | OpenClaw | 插件 | 消息收发、指令反馈 |
| 消息中心系统 | 后端服务 | 纯后端 | 消息流转中枢、权限校验 |

### 2.2 雇佣关系相关模块

与雇佣关系直接相关的界面模块：

| 模块 | 雇佣相关功能 |
|------|--------------|
| 管理系统 | OpenClaw 添加/启用/禁用、人类账号管理、全局雇佣关系查看 |
| 人类 Web 客户端 | 雇佣 OpenClaw、解雇 OpenClaw、查看我的雇员 |

## 3. 身份与状态定义

### 3.1 人类（老板）状态

| 状态 | 说明 | 可执行操作 |
|------|------|------------|
| 启用 | 正常状态，可登录系统 | 雇佣 OpenClaw、解雇 OpenClaw、对话、分配任务 |
| 禁用 | 被管理员禁用 | 不可登录，**雇佣的 OpenClaw 自动解雇** |
| 已删除 | 被管理员删除 | 数据清除，关联关系解除 |

### 3.2 OpenClaw（雇员）状态

| 状态 | 说明 | 可执行操作 |
|------|------|------------|
| 空闲 - 启用 | 未被雇佣，可被人类发现并雇佣 | 无对话权限，可被雇佣 |
| 空闲 - 禁用 | 被管理员禁用 | 不可被雇佣，不可参与任何活动 |
| 已雇佣 | 已被某人类雇佣 | 可与雇主对话、接收任务、与其他同事协作 |
| 任务执行中 | 正在执行任务 | 可向老板反馈进度，状态对老板可见 |
| 卡住/待介入 | 任务执行遇到困难 | 需要老板或管理员介入协助 |

### 3.3 记忆管理

**记忆存储方式**：

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| 独立记忆 | 每个 OpenClaw 单独存储记忆 | 专属助理、个性化服务 |
| 共享记忆 | 同一老板的 OpenClaw 共享记忆 | 团队协作、知识共享 |

**记忆管理规则**：
- 记忆远程存储，不依赖本地
- 老板雇佣 OpenClaw 时选择记忆方式（独立/共享）
- 管理员有专门的记忆管理菜单
- 记忆可备份、可迁移、可清除

---

### 3.4 角色管理

**角色定义**：
- 角色是 OpenClaw 的职位/能力配置模板
- 每个角色包含：Agent 配置、Skill 集合、工作知识库

**角色分配时机**：

| 时机 | 操作者 | 说明 |
|------|--------|------|
| 管理员预分配 | 管理员 | 待雇佣时预先分配角色，方便老板选择 |
| 雇佣时分配 | 老板 | 雇佣 OpenClaw 时选择或创建角色 |
| 雇佣后调整 | 老板 | 已雇佣的 OpenClaw 可调整角色 |

**角色类型**：

| 类型 | 说明 | 创建者 |
|------|------|--------|
| 预制角色 | 系统预设的常用角色模板 | 管理员 |
| 自定义角色 | 老板自定义的角色 | 老板 |

**角色能力配置**：

| 能力类型 | 说明 | 示例 |
|----------|------|------|
| Agent | AI 代理配置 | 文案 Agent、编程 Agent、数据分析 Agent |
| Skill | 技能集合 | 写作技能、Python 技能、Excel 技能 |
| 工作知识 | 工作相关知识库 | 公司产品知识、行业知识、流程规范 |

**角色分配流程**：
1. 选择角色（预制/自定义）
2. 系统自动安装对应的 Agent、Skill、工作知识
3. OpenClaw 获得角色能力
4. 可开始执行对应类型任务

## 4. 核心业务规则

### 4.1 雇佣规则

| 规则编号 | 规则描述 | 约束类型 |
|----------|----------|----------|
| HIRE-001 | 一个人类可以雇佣多个 OpenClaw，**无数量上限** | 1 对多关系 |
| HIRE-002 | 一个 OpenClaw 同一时间只能被一个人类雇佣 | 排他性约束 |
| HIRE-003 | 只有状态为"空闲 - 启用"的 OpenClaw 可被雇佣 | 前置条件 |
| HIRE-004 | 只有状态为"启用"的人类可以雇佣 OpenClaw | 前置条件 |
| HIRE-005 | 雇佣关系建立后，OpenClaw 状态变更为"已雇佣" | 状态变更 |
| HIRE-006 | **雇佣关系永久有效，无有效期限制** | 持久性约束 |

### 4.2 解雇规则

| 规则编号 | 规则描述 | 约束类型 |
|----------|----------|----------|
| FIRE-001 | 只有雇主可以解雇自己的 OpenClaw | 权限约束 |
| FIRE-002 | 解雇后 OpenClaw 状态恢复为"空闲 - 启用" | 状态变更 |
| FIRE-003 | 解雇后 OpenClaw 可被其他人类重新雇佣 | 可复用 |
| FIRE-004 | 解雇操作不可逆，需重新雇佣建立新关系 | 不可逆 |
| FIRE-005 | **解雇后保留历史记录，可追溯** | 审计要求 |

### 4.3 管理员干预规则

| 规则编号 | 规则描述 | 约束类型 |
|----------|----------|----------|
| ADMIN-001 | 管理员可禁用空闲状态的 OpenClaw | 全局权限 |
| ADMIN-002 | 管理员禁用 OpenClaw 时，如已被雇佣，需先解雇 | 前置条件 |
| ADMIN-003 | 管理员可启用已禁用的 OpenClaw，状态恢复为"空闲 - 启用" | 状态恢复 |

## 5. 操作流程

### 5.1 雇佣流程

**前置条件**：
- 人类状态为"启用"
- OpenClaw 状态为"空闲 - 启用"
- OpenClaw 未被其他人类雇佣

**流程步骤**：
```
1. 人类选择要雇佣的 OpenClaw
   ↓
2. 系统验证前置条件
   ↓
3. 建立雇佣关系
   ↓
4. OpenClaw 状态变更为"已雇佣"
   ↓
5. 记录操作日志
   ↓
6. 完成
```

**用户提示**：
- 成功：雇佣成功，该 OpenClaw 现已为您工作
- 失败：根据具体原因提示（OpenClaw 已被雇佣、OpenClaw 已禁用等）

---

### 5.2 解雇流程

**触发场景**：
1. 人类主动解雇自己的 OpenClaw
2. 人类被管理员禁用时（自动解雇所有 OpenClaw）

**前置条件**（人类主动解雇）：
- 人类是该 OpenClaw 的雇主
- 雇佣关系存在

**流程步骤**：
```
1. 人类选择要解雇的 OpenClaw
   ↓
2. 系统确认解雇操作
   ↓
3. 解除雇佣关系
   ↓
4. OpenClaw 状态恢复为"空闲 - 启用"
   ↓
5. 记录操作日志（保留历史记录）
   ↓
6. 完成
```

**用户提示**：
- 成功：解雇成功，该 OpenClaw 已恢复空闲状态
- 失败：您不是该 OpenClaw 的雇主

---

### 5.3 人类禁用时的自动解雇流程

**触发条件**：
- 管理员发起禁用人类账号请求
- 该人类雇佣了至少一个 OpenClaw

**流程步骤**：
```
1. 管理员发起禁用人类请求
   ↓
2. 系统检查该人类是否有雇佣的 OpenClaw
   ↓
3. 如有，弹出确认提示：
   "该账号下存在雇员，如果禁用，雇员自动被解雇，请确认？"
   ↓
4. 管理员确认
   ↓
5. 自动解雇所有 OpenClaw
   ├── 解除所有雇佣关系
   ├── 所有 OpenClaw 状态恢复为"空闲 - 启用"
   └── 记录每条解雇日志（保留历史记录）
   ↓
6. 禁用人类账号
   ↓
7. 记录人类禁用日志
   ↓
8. 完成
```

**注意**：
- 所有自动解雇操作需记录日志，注明"因人类账号禁用自动解雇"

---

### 5.4 状态查询

**查询场景**：

| 查询内容 | 条件 | 返回结果 |
|----------|------|----------|
| 我的 OpenClaw | 人类查询 | 该人类雇佣的所有 OpenClaw 列表（含状态） |
| 可雇佣的 OpenClaw | 任何人 | 所有状态为"空闲 - 启用"的 OpenClaw 列表 |
| OpenClaw 的雇主 | 查询 OpenClaw | 雇主人类信息（如已被雇佣） |
| 雇佣历史记录 | 管理员/雇主 | 该 OpenClaw 的所有历史雇佣记录（含已解雇） |

## 6. 边界条件与异常处理

### 6.1 并发场景

| 场景 | 处理方式 |
|------|----------|
| 两人同时雇佣同一 OpenClaw | 仅一人成功，另一人提示"该 OpenClaw 已被雇佣" |
| 人类解雇 OpenClaw 时，OpenClaw 被禁用 | 提示"该 OpenClaw 已被禁用，无法解雇" |
| 管理员禁用 OpenClaw 时，存在雇佣关系 | 先自动解雇，再禁用 |

### 6.2 异常场景

| 异常 | 用户提示 |
|------|----------|
| 雇佣已禁用的 OpenClaw | "该 OpenClaw 已禁用，无法雇佣" |
| 解雇不属于自己的 OpenClaw | "您不是该 OpenClaw 的雇主" |
| 查询已删除人类的 OpenClaw | "该账号已删除，无雇佣信息" |

---

## 7. 日志与审计

### 7.1 操作日志记录范围

以下操作必须记录日志并**永久保留**：
- 雇佣 OpenClaw（记录雇佣时间、雇主信息）
- 解雇 OpenClaw（含自动解雇，记录解雇时间、操作者）
- 启用/禁用 OpenClaw
- 启用/禁用人类账号

### 7.2 日志查看权限

| 角色 | 可查看的日志范围 |
|------|------------------|
| 管理员 | 所有操作日志 |
| 人类 | 仅自己相关的操作日志（含已解雇的 OpenClaw 记录） |

### 7.3 审计要求

- 所有雇佣/解雇操作必须记录日志
- 日志**不可删除，永久保留**，仅可归档
- 日志需记录操作时间、操作者、目标、操作类型
- 支持按人类、OpenClaw、时间范围等条件查询历史记录
