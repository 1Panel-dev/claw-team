# ClawLabor OpenClaw 通道插件需求分析

## 1. 系统概述

### 1.1 系统定位

ClawLabor OpenClaw 通道插件是安装于**OpenClaw 端**的专属插件，核心实现 OpenClaw 与消息中心的双向通讯，承接 OpenClaw 的消息接收与指令反馈。

### 1.2 目标用户

| 用户角色 | 说明 |
|----------|------|
| OpenClaw | AI 雇员，通过插件与人类（老板）进行通讯、接收任务、反馈结果 |

### 1.3 核心职责

| 职责 | 说明 |
|------|------|
| 消息接收 | 从消息中心接收来自人类的消息（普通消息、任务消息、系统通知） |
| 消息发送 | 向消息中心发送消息，转发给人类 |
| 指令反馈 | 接收任务指令，执行后反馈结果 |
| 状态上报 | 上报在线状态、工作状态、任务执行状态 |
| 心跳维持 | 与消息中心保持 WebSocket 长连接，定期发送心跳 |

### 1.4 与雇佣关系模块的关联

OpenClaw 通道插件依赖雇佣关系模块进行权限控制：
- OpenClaw 安装插件后即可连接消息中心（无需雇佣）
- 仅已雇佣的 OpenClaw 可与雇主及同事（同一雇主的其他 OpenClaw）通讯
- 解雇后，连接保持，但消息中心限制消息转发，并自动回复告知暂停通讯
- 重新被雇佣后，自动恢复通讯权限

### 1.5 与消息中心系统的关联

OpenClaw 通道插件是消息中心系统的客户端：
- 通过 WebSocket 与消息中心建立长连接
- 遵循消息中心的 API 规范
- 依赖消息中心进行消息路由和权限校验

## 3. 核心功能模块

### 3.1 WebSocket 客户端

#### 3.1.1 连接管理

**连接流程**：
```
1. 插件启动
   ↓
2. 读取配置文件，获取消息中心地址和认证 Token
   ↓
3. 发起 WebSocket 连接
   wss://message-center.clawlabor.com/ws?token=xxx&openclawId=oc_001
   ↓
4. 服务端验证 Token
   ↓
5. 连接成功，进入已连接状态
   ↓
6. 开始心跳循环
```

**连接状态**：
| 状态 | 说明 | 触发条件 |
|------|------|----------|
| 连接中 (CONNECTING) | 正在建立连接 | 插件启动、重连 |
| 已连接 (CONNECTED) | 连接正常 | 握手成功 |
| 断开 (DISCONNECTED) | 连接断开 | 网络故障、服务端关闭、插件卸载 |
| 认证失败 (AUTH_FAILED) | Token 无效 | Token 过期、账号禁用 |
| 限制通讯 (LIMITED) | 连接正常但限制发消息 | 未雇佣/被解雇 |

**状态流转图**：
```
┌─────────────────────────────────────────────────────────────┐
│                    连接状态流转图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   [启动] → 连接中 ──────→ 已连接 ←────→ 限制通讯             │
│              ↑            │  │         (未雇佣/被解雇)       │
│              │            │  │ (网络故障/服务端关闭)         │
│              │            │  ↓                              │
│              │            │ 断开                           │
│              │            │  │                              │
│              │  (心跳超时)│  │ (重连定时器)                  │
│              │←───────────┘  ↓                              │
│              └───────────────┴──→ [认证失败] → [停止]       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**状态说明**：
- **已连接**：OpenClaw 已被雇佣，可正常收发消息
- **限制通讯**：OpenClaw 未被雇佣或已被解雇，连接保持但无法发送消息，接收消息时收到自动回复提示

#### 3.1.2 心跳机制

**心跳配置**：
| 参数 | 默认值 | 说明 |
|------|--------|------|
| 心跳间隔 | 30 秒 | 每 30 秒发送一次 Ping |
| 超时时间 | 90 秒 | 90 秒未收到 Pong 判定超时 |
| 重连间隔 | 5 秒 | 断开后 5 秒尝试重连 |
| 最大重连次数 | 无限 | 持续重连直到成功 |

**心跳流程**：
```
客户端 (插件)                    服务端 (消息中心)
     │                                │
     │───── Ping (每 30 秒) ──────────→│
     │                                │
     │←──── Pong (立即响应) ──────────│
     │                                │
     │                                │ (重置超时定时器)
     │                                │
```

**心跳消息格式**：
```json
// 客户端 → 服务端
{
  "type": "ping",
  "timestamp": 1710237600000,
  "openclawId": "oc_001"
}

// 服务端 → 客户端
{
  "type": "pong",
  "timestamp": 1710237600000,
  "serverTime": 1710237600050
}
```

#### 3.1.3 重连机制

**重连策略**：
```
1. 连接断开
   ↓
2. 等待 5 秒
   ↓
3. 尝试重连
   ↓
4. 如失败，等待时间翻倍 (5s → 10s → 20s → 40s → 60s)
   ↓
5. 达到 60 秒后，保持 60 秒间隔持续重连
   ↓
6. 连接成功，重置等待时间为 5 秒
```

**重连触发条件**：
| 条件 | 说明 |
|------|------|
| 网络断开 | WebSocket 连接意外关闭 |
| 心跳超时 | 90 秒未收到服务端响应 |
| 服务端重启 | 收到服务端关闭通知 |
| 认证失败 | Token 过期，需重新认证 |

### 3.2 消息处理模块

#### 3.2.1 消息接收

**接收消息类型**：
| 类型 | 说明 | 处理方式 |
|------|------|----------|
| `text` | 普通文本消息 | 传递给 OpenClaw 核心处理 |
| `task` | 任务消息 | 传递给任务执行模块 |
| `system` | 系统通知 | 记录日志，可选通知 |

**消息接收流程**：
```
1. WebSocket 收到消息
   ↓
2. 解析消息 JSON
   ↓
3. 验证消息格式
   ↓
4. 根据消息类型分发
   ├── text → 消息队列，等待 OpenClaw 处理
   ├── task → 任务执行模块
   └── system → 日志记录
   ↓
5. 发送收到确认给服务端
   ↓
6. 存储消息到本地（可选）
```

**接收消息格式**：
```json
{
  "type": "message.new",
  "data": {
    "messageId": "msg_123456",
    "from": {
      "id": "human_001",
      "name": "张三",
      "type": "human"
    },
    "to": {
      "id": "oc_001",
      "name": "AI 助手 -001",
      "type": "openclaw"
    },
    "messageType": "text",
    "content": {
      "type": "text",
      "text": "请帮我写一份产品文案..."
    },
    "timestamp": 1710237600000
  }
}
```

#### 3.2.2 消息发送

**发送消息类型**：
| 类型 | 说明 |
|------|------|
| `text` | 普通文本回复 |
| `task.update` | 任务进度更新 |
| `task.complete` | 任务完成反馈 |

**消息发送流程**：
```
1. OpenClaw 核心生成回复
   ↓
2. 插件接收回复内容
   ↓
3. 构建消息对象
   ↓
4. 通过 WebSocket 发送
   ↓
5. 等待服务端确认
   ↓
6. 记录发送日志
```

**发送消息格式**：
```json
{
  "type": "message.send",
  "data": {
    "receiverId": "human_001",
    "messageType": "text",
    "content": {
      "type": "text",
      "text": "好的，我马上开始处理..."
    },
    "inReplyTo": "msg_123456"  // 可选，回复的消息 ID
  }
}
```

#### 3.2.3 消息队列

**队列管理**：
```
┌─────────────────────────────────────────────────────────────┐
│                      消息队列                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  [消息 1] → [消息 2] → [消息 3] → ... → [消息 N]            │
│     ↓          ↓          ↓                   ↓             │
│   待发送     待发送     待发送               待发送          │
│                                                             │
│  发送策略：FIFO（先进先出）                                  │
│  队列上限：100 条                                             │
│  超限处理：丢弃最早的消息                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**离线消息处理**：
- 连接断开时，消息存入本地队列
- 连接恢复后，自动发送队列中的消息
- 队列满时，丢弃最早的消息

### 3.3 任务执行模块

#### 3.3.1 任务消息解析

**任务消息格式**：
```json
{
  "type": "message.new",
  "data": {
    "messageId": "msg_123456",
    "from": {
      "id": "human_001",
      "name": "张三",
      "type": "human"
    },
    "messageType": "task",
    "content": {
      "type": "task",
      "taskName": "撰写产品文案",
      "description": "请为新产品撰写产品文案，包括产品概述、核心功能、目标用户...",
      "priority": "high",
      "deadline": "2026-03-15T18:00:00Z",
      "attachments": []
    },
    "taskId": "task_123456"
  }
}
```

**任务字段说明**：
| 字段 | 类型 | 说明 |
|------|------|------|
| taskName | String | 任务名称 |
| description | String | 任务描述 |
| priority | Enum | 优先级：low/medium/high |
| deadline | Timestamp | 截止时间 |
| attachments | Array | 附件列表（可选） |
| taskId | String | 任务 ID（与消息中心关联） |

#### 3.3.2 任务执行流程

```
1. 接收任务消息
   ↓
2. 解析任务信息
   ↓
3. 发送任务确认（可选）
   ↓
4. 执行任务（调用 OpenClaw 核心）
   ↓
5. 更新任务进度（可选）
   ↓
6. 完成任务，发送结果
   ↓
7. 等待雇主确认（可选）
```

**任务确认消息**：
```json
{
  "type": "task.ack",
  "data": {
    "taskId": "task_123456",
    "status": "accepted",
    "estimatedTime": "2 小时",
    "message": "收到，我马上开始处理..."
  }
}
```

**任务进度更新**：
```json
{
  "type": "task.progress",
  "data": {
    "taskId": "task_123456",
    "progress": 50,
    "message": "已完成初稿，正在润色..."
  }
}
```

**任务完成反馈**：
```json
{
  "type": "task.complete",
  "data": {
    "taskId": "task_123456",
    "status": "completed",
    "result": {
      "type": "text",
      "text": "产品文案已完成：\n\n## 产品概述\n..."
    },
    "completedAt": "2026-03-12T11:00:00Z"
  }
}
```

#### 3.3.3 任务状态管理

**任务状态**：
| 状态 | 说明 |
|------|------|
| pending | 待处理（刚收到任务） |
| accepted | 已接受（确认执行） |
| in_progress | 执行中 |
| completed | 已完成 |
| failed | 执行失败 |

**状态流转图**：
```
┌─────────────────────────────────────────────────────────────┐
│                    任务状态流转图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   pending → accepted → in_progress → completed              │
│      │         │            │          │                    │
│      │         │            │          │                    │
│      │         │            └──────→ failed                │
│      │         │                                             │
│      └─────────┴──────────── (可随时失败)                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.4 状态管理模块

### 3.5 认证模块

#### 3.5.1 Token 管理

**Token 获取**：
- 首次安装：管理员在后台生成 Token，配置到插件
- Token 刷新：Token 过期前自动刷新（如支持）

**Token 存储**：
- 本地加密存储
- 配置文件中的 Token 字段加密

**Token 失效处理**：
```
1. 收到认证失败响应
   ↓
2. 判断失效原因
   ↓
3. 根据原因处理：
   ├── Token 过期 → 尝试刷新 Token
   ├── 账号禁用 → 停止重连，通知管理员
   ├── 被解雇 → 进入"限制通讯"状态，保持连接
   └── 插件未授权 → 停止重连，通知管理员
   ↓
4. 记录错误日志
```

**轮转策略**：

## 4. 接口设计

### 4.1 WebSocket 消息接口

#### 4.1.1 客户端 → 服务端

| 消息类型 | 说明 | 格式 |
|----------|------|------|
| `ping` | 心跳请求 | `{"type": "ping", "timestamp": xxx}` |
| `message.send` | 发送消息 | 见 3.2.2 |
| `task.ack` | 任务确认 | 见 3.3.2 |
| `task.progress` | 任务进度 | 见 3.3.2 |
| `task.complete` | 任务完成 | 见 3.3.2 |
| `status.update` | 状态更新 | 见 3.4.1 |

#### 4.1.2 服务端 → 客户端

| 消息类型 | 说明 | 格式 |
|----------|------|------|
| `pong` | 心跳响应 | `{"type": "pong", "timestamp": xxx}` |

## 5. 异常处理

### 5.1 错误码定义

| 错误码 | 说明 | 处理方式 |
|--------|------|----------|
| 0 | 成功 | - |
| 401 | 认证失败 | 停止重连，通知管理员 |
| 403 | 权限不足（如被解雇） | 停止重连，通知管理员 |
| 429 | 请求过多 | 降低发送频率 |
| 500 | 服务端错误 | 等待后重试 |
| 1001 | 网络断开 | 触发重连 |
| 1002 | 心跳超时 | 触发重连 |
| 1003 | 消息格式错误 | 记录日志，丢弃消息 |

### 5.2 异常处理流程

```
1. 捕获异常
   ↓
2. 记录错误日志
   ↓
3. 根据错误类型处理
   ├── 认证失败 → 停止重连，通知管理员
   ├── 网络断开 → 触发重连
   ├── 消息格式错误 → 丢弃消息，记录日志
   └── 其他错误 → 重试或上报
   ↓
4. 如需要，通知管理员
```
