## 一、系统概述

### 1.1 系统定位

管理系统是 1Claw 的系统管理员专属后台，用于：
- 人类账号管理（注册、启用/禁用、删除）
- OpenClaw 管理（添加、启用/禁用、全局查看）
- 雇佣关系管理（全局查看、历史记录）
- 系统日志与审计
- 角色与技能管理
- 系统配置

### 1.2 架构模式

```
┌─────────────────┐      HTTP/REST      ┌─────────────────┐
│  管理系统前端    │ ───────────────────→ │  管理系统后端    │
│  (Vue 3 + TS)   │ ←─────────────────── │  (FastAPI)      │
└─────────────────┘                     └────────┬────────┘
                                                  │
                                                  │ 直连
                                                  ▼
                                         ┌─────────────────┐
                                         │   PostgreSQL    │
                                         │   (数据库)      │
                                         └─────────────────┘
```

**关键特点**：
- **前后端分离**：前端与后端独立部署、独立开发
- **直连数据库**：管理系统后端直接连接 PostgreSQL（不经过调度中心）
- **RESTful API**：标准化接口设计，支持文档自动生成

## 四、模块化设计

### 4.1 设计原则

**目标**：让多个开发者可以并行开发，互不影响

| 原则 | 说明 |
|------|------|
| **高内聚** | 每个模块只负责一个业务领域 |
| **低耦合** | 模块间通过 API/接口通讯，不直接依赖实现 |
| **单一职责** | 每个文件/类只做一件事 |
| **可替换** | 模块可独立测试、独立替换 |

### 4.2 模块划分

#### 前端模块（按业务功能）

| 模块 | 目录 | 依赖 |
|------|------|------|
| 认证模块 | `src/views/auth/` | 无 |
| 仪表盘 | `src/views/dashboard/` | 无 |
| 人类账号 | `src/views/humans/` | API: `api/humans.ts` |
| OpenClaw | `src/views/openclaws/` | API: `api/openclaws.ts` |
| 雇佣关系 | `src/views/employments/` | API: `api/employments.ts` |
| 系统日志 | `src/views/logs/` | API: `api/logs.ts` |
| 角色管理 | `src/views/roles/` | API: `api/roles.ts` |

**开发规则**：
- 每个模块有独立的 `views/`、`api/`、`types/` 文件
- 模块间不直接引用对方的内部实现
- 公共组件放入 `components/common/` 或 `components/business/`
- 共享状态放入 Pinia Store

#### 后端模块（按业务功能）

| 模块 | 目录 | 依赖 |
|------|------|------|
| 认证 | `app/api/deps.py` | `app/utils/auth.py` |
| 人类账号 | `app/api/humans.py` + `app/services/human_service.py` | `app/models/human.py` |
| OpenClaw | `app/api/openclaws.py` + `app/services/openclaw_service.py` | `app/models/openclaw.py` |
| 雇佣关系 | `app/api/employments.py` + `app/services/employment_service.py` | `app/models/employment.py` |
| 系统日志 | `app/api/logs.py` + `app/services/log_service.py` | `app/models/log.py` |
| 角色管理 | `app/api/roles.py` + `app/services/role_service.py` | `app/models/role.py` |

**开发规则**：
- API 层只负责请求处理，业务逻辑放入 Service 层
- Service 层通过 ORM 模型访问数据库
- 模块间不直接调用对方的 Service，通过 API 或事件通讯
- 共享工具放入 `app/utils/`

## 五、开发规范（概要）

> 详细规范文档由团队根据本框架编写，以下为基本要求。

### 5.1 代码规范

#### 前端（TypeScript/Vue）

| 项目 | 要求 |
|------|------|
| 代码风格 | ESLint + Prettier 自动格式化 |
| 命名规范 | 组件 PascalCase，文件 kebab-case |
| 类型系统 | 严格模式，禁止 `any` |
| 组件设计 | 组合式 API，逻辑复用使用 composables |
| 注释要求 | 公共函数、复杂逻辑必须有注释 |

#### 后端（Python）

| 项目 | 要求 |
|------|------|
| 代码风格 | black 自动格式化 |
| 代码检查 | flake8 + mypy 静态类型检查 |
| 命名规范 | 函数/变量 snake_case，类 PascalCase |
| 类型提示 | 所有函数必须有类型注解 |
| 文档字符串 | 所有公共函数必须有 docstring |

### 5.2 提交规范

**格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**type 类型**：
| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档更新 |
| `style` | 代码格式（不影响功能） |
| `refactor` | 重构 |
| `test` | 测试相关 |
| `chore` | 构建/工具链/配置 |

**scope 范围**：按模块命名（humans、openclaws、employments、logs、roles 等）

### 5.3 分支策略

```
main                    # 生产分支（受保护）
├── develop             # 开发分支
│   ├── feature/<module>-<desc>  # 功能分支
│   ├── bugfix/<desc>            # 修复分支
│   └── release/v<version>       # 发布分支
```

**规则**：
| 分支 | 命名 | 来源 | 合并目标 | 说明 |
|------|------|------|----------|------|
| 功能分支 | `feature/<module>-<desc>` | develop | develop | 新功能开发 |
| 修复分支 | `bugfix/<desc>` | develop | develop | Bug 修复 |
| 发布分支 | `release/v<version>` | develop | main + develop | 版本发布 |
| 热修复 | `hotfix/<desc>` | main | main + develop | 生产环境紧急修复 |

### 5.4 设计规范

#### API 设计规范

| 项目 | 要求 |
|------|------|
| URL 命名 | 小写 + 连字符，复数形式 |
| HTTP 方法 | GET(查询)、POST(创建)、PUT(全量更新)、PATCH(部分更新)、DELETE(删除) |
| 响应格式 | 统一封装 |
| 错误处理 | 统一异常处理 |
| 认证方式 | JWT Token |
| 文档生成 | FastAPI 自动生成 |

#### 前端组件设计规范

| 项目 | 要求 |
|------|------|
| 组件拆分 | 单一职责 |
| Props 定义 | 使用 TypeScript 接口 |
| 事件命名 | `onXxx` 或 `update:xxx` |
| 样式隔离 | 使用 `scoped` 或 CSS Modules |

#### 数据库设计规范

| 项目 | 要求 |
|------|------|
| 表命名 | 小写 + 下划线，复数形式 |
| 主键 | 统一使用 `id`（自增） |
| 时间字段 | `created_at`、`updated_at` |
| 软删除 | 使用 `deleted_at` |
| 迁移管理 | 使用 Alembic |

## 六、开发环境

### 6.1 开发环境（本机启动）

**设计原则**：
- 基础设施（PostgreSQL、Redis）由外部提供，无需本机启动
- 本机只需启动应用服务（后端 + 前端）
- 从统一配置文件读取配置（与生产环境一致）

**启动方式**：
- 后端：`uvicorn --reload`（热重载）
- 前端：`npm run dev`（热重载）
- 独立端口，避免冲突

**说明**：
- ✅ **无需 Docker**：本机只运行应用代码
- ✅ **热重载**：方便开发调试
- ✅ **配置一致**：与生产环境使用相同的配置格式
- ✅ **基础设施外置**：PostgreSQL、Redis 由外部提供

## 八、部署说明

### 8.1 部署方案

**当前方案（方案一）**：管理系统与调度中心、Web 客户端打包到同一个 Docker 容器中

- 前端构建为静态文件，由 Traefik 统一服务
- 后端由 Supervisor 管理，与调度中心后端同容器运行
- 数据库 PostgreSQL 独立部署（容器外或单独容器）
- **统一配置文件**：从外部挂载的 config.yaml 读取配置

**未来方案（方案二，预留）**：管理系统独立部署

- 管理系统前端 + 后端 + Traefik 独立 Docker 容器
- 与核心业务（调度中心 + Web 客户端）分离
- 适用于独立升级、独立访问控制场景

### 8.2 配置管理

**配置方式**：统一配置文件（YAML 格式）

**配置文件位置**：
- 宿主机：`/opt/1claw/config/config.yaml`
- 容器内：`/config/config.yaml`（只读挂载）

**配置文件内容**：
- 应用配置（环境、日志级别等）
- 数据库配置
- Redis 配置
- JWT 配置
- 存储配置
- CORS 配置
- 服务端口配置
- 其他子系统配置

**配置更新流程**：
1. 编辑配置文件
2. 重启服务
3. 配置生效（无需重建镜像）

**详细说明**：参见 [01-开发框架.md](./01-开发框架.md) §6.4 配置管理方案

### 8.3 前端部署

**构建方式**：
- 命令：`npm run build`
- 输出：静态文件目录
- 部署：复制到 Docker 镜像，由 Traefik 服务

### 8.4 后端部署

**启动方式**：
- 进程管理：Supervisor 统一管理
- 启动命令：uvicorn 或 Gunicorn
- 多进程：支持多 worker 模式

### 8.5 Docker 部署（方案一）

**构建方式**：
- 多阶段构建：分别构建前端和后端
- 前端：构建为静态文件
- 后端：安装依赖，复制代码
- 入口：配置 Supervisor，启动所有服务

**部署目录结构**：
```
/opt/1claw/
├── docker-compose.yml
├── .env                    # 敏感信息
├── config/
│   └── config.yaml         # 统一配置文件 ⭐
├── data/                   # 应用数据
├── logs/                   # 日志文件
└── postgres-data/          # PostgreSQL 数据
```

**启动命令**：
```bash
cd /opt/1claw
docker-compose up -d
```

**更新服务**：
```bash
# 更新镜像
docker-compose pull
docker-compose up -d --force-recreate

# 或更新配置
vim /opt/1claw/config/config.yaml
docker-compose restart
```
