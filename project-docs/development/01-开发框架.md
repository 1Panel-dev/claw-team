## 二、技术栈选型

### 2.1 前端技术栈

| 子系统 | 框架 | 构建工具 | 语言 | UI 组件库 | 状态管理 |
|--------|------|----------|------|-----------|----------|
| 人类 Web 客户端 | Vue 3 | Vite | TypeScript | Element Plus | Pinia |

**选型理由**：
- Vue 3：团队熟悉度高，生态完善，适合中后台应用
- Vite：构建速度快，开发体验好
- TypeScript：类型安全，减少运行时错误
- Element Plus：成熟的企业级 UI 组件库，组件丰富，文档完善
- Pinia：Vue 3 官方推荐的状态管理库

### 2.2 后端技术栈

| 子系统 | 语言 | Web 框架 | ORM | 认证 |
|--------|------|----------|-----|------|
| 调度中心系统 | Python 3.11+ | FastAPI | SQLAlchemy 2.0（支持 SQLite/PostgreSQL） | JWT（可选） |

**选型理由**：
- Python：开发效率高，AI 生态丰富（与 OpenClaw 集成友好）
- FastAPI：异步支持好，自动生成 API 文档，性能优秀
- SQLAlchemy 2.0：成熟的 ORM，支持异步，类型提示完善，完美支持 SQLite
- JWT：可选，单用户模式可简化认证

### 2.3 数据库选型

| 数据库 | 用途 | 版本 |
|--------|------|------|
| SQLite | 主数据库（默认） | 3.40+ |
| PostgreSQL | 主数据库（可选） | 15+ |

**SQLite 选型理由**：
- 零配置，开箱即用
- 无需 Docker 或外部服务
- 单用户场景性能足够
- 便携，复制文件即可迁移
- SQLAlchemy 完美支持

**PostgreSQL 选型理由**：
- 开源免费，功能强大
- 支持 JSONB 字段
- 并发性能更好
- SQLAlchemy 同样支持

**ORM**：SQLAlchemy 2.0（支持 SQLite 和 PostgreSQL）

### 2.4 中间件选型

| 中间件 | 用途 | 选型 |
|--------|------|------|
| 消息队列 | 异步任务、通知发送 | 内存队列 |
| WebSocket 服务 | 实时通讯 | FastAPI WebSocket（原生） |
| 文件存储 | 工作空间文件存储 | 本地存储（v1）/ MinIO（v2） |
| API 网关 | 请求路由、限流、HTTPS | Nginx |
| 定时任务 | 后台定时任务、周期性检查 | APScheduler |

**文件存储策略说明**：
- **v1 默认**: 本地存储（简单部署，无需额外依赖）
- **v1 可选**: MinIO（生产环境，支持分布式/集群）
- **实现方式**: 通过配置文件切换存储后端，代码层使用统一接口
- **配置项**: `STORAGE_BACKEND = local | minio`
- **优势**: 
  - 小规模部署：零依赖，开箱即用
  - 大规模部署：无缝切换到 MinIO，支持 S3 协议
  - 未来扩展：可轻松迁移到其他 S3 兼容存储（AWS S3, 阿里云 OSS 等）

**消息队列选型说明**：
- 选用 **内存队列**（个人使用，消息量少）
- 无需额外依赖，零配置
- 足够支撑个人用户的异步任务和通知需求
- 未来如需持久化，再考虑引入 Redis

**定时任务选型说明**：
- 选用 **APScheduler**（轻量级异步调度器）
- 与 FastAPI 异步架构完美集成
- 无需额外依赖（Celery Beat 需要 Celery Worker）
- 支持多种触发器：interval（间隔）、cron（cron 表达式）、date（一次性）
- **典型场景**：
  - ⏰ 截止时间提醒（任务到期前 1 小时/1 天自动通知）
  - 💓 心跳检查（检查 OpenClaw 通道在线状态）
  - 🧹 数据清理（清理过期临时文件、日志）
  - 📊 状态扫描（逾期任务自动标记）
- **注意**：此"定时任务"与"任务管理（Task Management）"是两个独立功能
  - 定时任务 = 后台定时执行的系统任务（类似 Cron Jobs）
  - 任务管理 = 业务任务系统（类似 TAPD/Jira，用于分配和追踪工作任务）

## 三、数据库设计规范

### 3.1 字符集与编码

| 配置项 | 要求 | 说明 |
|--------|------|------|
| **字符集** | `UTF8` | 支持多语言，包括中文 |
| **排序规则** | `en_US.UTF-8` 或 `zh_CN.UTF-8` | 根据部署地区选择 |
| **时区** | `UTC` | 统一使用 UTC 时区，应用层转换 |

---

### 3.2 命名规范

#### 表命名

| 规则 | 示例 | 说明 |
|------|------|------|
| 小写字母 + 下划线 | `humans`, `task_comments` | 统一小写 |
| 复数形式 | `users`, `tasks` | 表示集合 |
| 语义清晰 | `employment_history` | 避免缩写 |
| 避免保留字 | ❌ `order`, ✅ `task_order` | 防止 SQL 冲突 |

#### 字段命名

| 规则 | 示例 | 说明 |
|------|------|------|
| 小写字母 + 下划线 | `created_at`, `user_id` | 统一小写 |
| 主键统一 | `id` | 所有表主键都叫 `id` |
| 外键命名 | `<表名>_id` | 如 `user_id`, `task_id` |
| 布尔字段 | `is_*`, `has_*`, `can_*` | 如 `is_active`, `has_permission` |
| 时间字段 | `*_at` | 如 `created_at`, `updated_at` |
| 避免保留字 | ❌ `order`, ✅ `sort_order` | 防止 SQL 冲突 |

#### 索引命名

| 索引类型 | 命名格式 | 示例 |
|---------|---------|------|
| 主键索引 | `pk_<表名>` | `pk_users` |
| 外键索引 | `fk_<表名>_<字段名>` | `fk_tasks_user_id` |
| 唯一索引 | `uk_<表名>_<字段名>` | `uk_users_email` |
| 普通索引 | `idx_<表名>_<字段名>` | `idx_tasks_status` |

#### 约束命名

| 约束类型 | 命名格式 | 示例 |
|---------|---------|------|
| 唯一约束 | `uk_<表名>_<字段名>` | `uk_users_email` |
| 检查约束 | `ck_<表名>_<字段名>` | `ck_users_status` |
| 外键约束 | `fk_<表名>_<字段名>` | `fk_tasks_user_id` |

---

### 3.3 字段类型规范

#### 数值类型

| 类型 | 用途 | 说明 |
|------|------|------|
| `INTEGER` | 主键 ID | 自增整数（SQLite: `AUTOINCREMENT`, PostgreSQL: `SERIAL`） |
| `BIGINT` | 大数值 | 如文件字节数、配额 |
| `DECIMAL` | 精确数值 | 如金额（避免浮点误差） |
| `BOOLEAN` | 布尔值 | `TRUE`/`FALSE`（SQLite: `INTEGER` 0/1） |

#### 字符串类型

| 类型 | 用途 | 说明 |
|------|------|------|
| `VARCHAR(n)` | 定长字符串 | 如用户名 `VARCHAR(50)` |
| `TEXT` | 长文本 | 如描述、内容 |
| `CHAR(n)` | 固定长度 | 如状态码、类型标识 |

#### 时间类型

| 类型 | 用途 | 说明 |
|------|------|------|
| `TIMESTAMP` | 日期时间 | 统一使用 `TIMESTAMP` |
| `DATE` | 仅日期 | 如生日、纪念日 |

#### 特殊类型

| 类型 | 用途 | 说明 |
|------|------|------|
| `JSON` / `JSONB` | 结构化数据 | 如配置、元数据（SQLite: `TEXT`, PostgreSQL: `JSONB`） |
| `ARRAY` | 数组 | 如标签列表（仅 PostgreSQL 支持） |

---

### 3.4 ID 设计规范

#### 方案选择

| 方案 | 类型 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **自增整数** | `INTEGER` | 简单、性能好、有序 | 可预测、分布式不友好 | 单数据库系统 |
| **UUID** | `TEXT/UUID` | 全局唯一、分布式友好 | 性能稍差、无序 | 分布式系统 |

**1Claw 推荐**：`INTEGER`（自增整数）

**理由**：
- 1Claw 是私有部署，单数据库场景
- 性能好，索引效率高
- 便于调试和排查问题

#### ID 字段要求

```
- 所有表必须有 `id` 字段
- 类型统一为 `INTEGER`
- 主键约束，自增
```

### 3.5 时间字段规范

#### 必填字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `created_at` | `TIMESTAMP` | 创建时间，必填 |
| `updated_at` | `TIMESTAMP` | 更新时间，必填 |

#### 可选字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `deleted_at` | `TIMESTAMP` | 软删除时间，NULL 表示未删除 |
| `*_at` | `TIMESTAMP` | 业务相关时间，如 `started_at`, `ended_at` |

#### 默认值

```sql
-- 推荐默认值
created_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP
deleted_at      TIMESTAMP NULL
```

#### 时区处理

```
- 数据库统一存储 UTC 时间
- 应用层负责转换为本地时区
- 不存储时区信息（简化查询）
```

### 3.7 软删除规范

#### 设计要求

```
- 核心业务表支持软删除
- 使用 `deleted_at` 字段标记
- NULL 表示未删除，TIMESTAMP 表示删除时间
- 查询时默认过滤已删除数据
```

#### 适用表

| 表类型 | 软删除 | 说明 |
|--------|-------|------|
| 用户表 | ✅ | 保留历史数据 |
| 业务表 | ✅ | 如任务、消息 |
| 日志表 | ❌ | 直接物理删除 |
| 历史表 | ❌ | 永久保留 |

### 3.8 数据完整性

#### 约束类型

| 约束 | 说明 | 示例 |
|------|------|------|
| `NOT NULL` | 非空约束 | 必填字段 |
| `UNIQUE` | 唯一约束 | 邮箱、用户名 |
| `PRIMARY KEY` | 主键约束 | id 字段 |
| `FOREIGN KEY` | 外键约束 | 关联关系 |
| `CHECK` | 检查约束 | 状态值范围 |

#### 外键规范

```
- 外键字段必须建立索引
- 删除策略：优先使用软删除，避免级联删除
- 更新策略：禁止更新主键
- 空值处理：根据业务决定是否允许 NULL
```

## 四、API 设计规范

### 4.1 RESTful API 规范

**URL 命名**：
- 使用小写字母和连字符：`/api/task-management`
- 资源名使用复数：`/api/tasks`, `/api/users`
- 嵌套资源：`/api/tasks/:id/comments`

**HTTP 方法**：
| 方法 | 用途 |
|------|------|
| GET | 获取资源 |
| POST | 创建资源 |
| PUT | 更新资源（全量） |
| PATCH | 更新资源（部分） |
| DELETE | 删除资源 |

**响应格式**：
```json
{
  "success": true,
  "data": { ... },
  "message": "操作成功",
  "timestamp": "2026-03-16T10:00:00Z"
}
```

**错误响应**：
```json
{
  "success": false,
  "error": {
    "code": "TASK_NOT_FOUND",
    "message": "任务不存在",
    "details": { ... }
  },
  "timestamp": "2026-03-16T10:00:00Z"
}
```

### 4.2 WebSocket 协议 (FastAPI 原生实现)

**连接建立**：
```
# 人类客户端
ws://host/ws/human/{human_id}?token=<JWT_TOKEN>

# OpenClaw 通道
ws://host/ws/openclaw/{openclaw_id}?token=<JWT_TOKEN>
```

**消息格式**：
```json
{
  "type": "message|task|notification|system",
  "action": "send|receive|update",
  "data": { ... },
  "timestamp": "2026-03-16T10:00:00Z"
}
```

**心跳机制**：
```json
// 客户端 → 服务端
{ "type": "ping", "timestamp": "2026-03-16T10:00:00Z" }

// 服务端 → 客户端
{ "type": "pong", "timestamp": "2026-03-16T10:00:00Z" }
```

**选型说明**：
- 采用 **FastAPI 原生 WebSocket**，无需 Socket.IO 额外依赖
- 私有部署环境，网络可控，无需轮询降级
- 移动端支持：各平台原生 WebSocket API 即可（iOS/Android/React Native/Flutter）
- 重连逻辑：客户端自行实现（简单指数退避策略）

## 六、部署架构

### 6.1 开发环境（本机启动）

**设计原则**：
- 基础设施（Nginx）由外部提供，无需本机启动
- 本机只需启动各应用服务（后端 + 前端）
- Channel 插件在 OpenClaw 环境中运行，单独部署
- 数据库：SQLite（本地文件）
- 消息队列：内存队列（无需外部服务）

**架构示意**：
```
开发主机
├── 调度中心后端（uvicorn :8001）
├── 人类 Web 客户端（npm dev :3000）
    │
    └──→ 连接外部基础设施
         ├── SQLite（本地文件）
         └── Nginx（可选）
```

**启动方式**：
- 后端：`uvicorn --reload`（热重载）
- 前端：`npm run dev`（热重载）
- 各服务独立端口，避免冲突

**说明**：
- ✅ **无需 Docker**：本机只运行应用代码
- ✅ **热重载**：方便开发调试
- ✅ **独立端口**：避免端口冲突
- ✅ **基础设施外置**：SQLite 本地文件，或外部 PostgreSQL/Nginx
- ⚠️ **Channel 插件**：需要在 OpenClaw 环境中运行，单独部署

---

### 6.2 生产环境（方案一：单 Docker 容器）

**当前方案**：所有子系统打包到一个 Docker 容器中启动

**架构示意**：
```
Docker Container
├── Nginx（入口，80/443）
│   ├── web-client（静态文件）
├── scheduler-server（uvicorn :8001）
└── Supervisor（进程管理）
    │
    └──→ 连接外部基础设施
         └── SQLite（容器内）或 PostgreSQL
```

**构建方式**：
- 多阶段构建：分别构建前端和后端
- 前端：构建为静态文件，由 Nginx 服务
- 后端：由 Supervisor 统一管理进程
- 入口：Nginx 统一处理 HTTPS 和路由

**优势**：
- ✅ 部署简单：一个容器包含所有服务
- ✅ 版本一致：所有子系统版本绑定，避免兼容性问题
- ✅ 维护方便：升级时只需更新一个镜像
- ✅ 资源集中：共享容器内资源

**劣势**：
- ⚠️ 容器体积较大
- ⚠️ 单个服务更新需要重建整个镜像
- ⚠️ 扩展性较差（无法单独扩展某个服务）

---

### 6.3 配置管理方案

**设计原则**：
- **统一配置文件**：所有子系统共用一个配置文件
- **外部挂载**：配置文件通过 Docker 卷挂载，修改无需重建镜像
- **配置与代码分离**：配置文件不提交到 Git
- **配置验证**：启动时验证必要配置项

**配置文件格式**：YAML

**配置文件内容**：
- 应用配置（环境、日志级别等）
- 数据库配置（SQLite 路径或 PostgreSQL 连接）
- JWT 配置（可选）
- 存储配置（本地/MinIO）
- CORS 配置
- 服务端口配置
- WebSocket 配置
- 任务调度配置

**配置文件位置**：
```
宿主机：/opt/1claw/config/config.yaml
容器内：/config/config.yaml（只读挂载）
```

**部署目录结构**：
```
/opt/1claw/
├── docker-compose.yml
├── .env                    # 敏感信息（DB_PASSWORD 等）
├── config/
│   └── config.yaml         # 统一配置文件 ⭐
├── data/                   # 应用数据
│   └── 1claw.db            # SQLite 数据库（默认）
├── logs/                   # 日志文件
│   └── scheduler.log
└── postgres-data/          # PostgreSQL 数据（可选）
```

**配置更新流程**：
1. 编辑配置文件
2. 重启服务
3. 配置生效（无需重建镜像）

**优势**：
- ✅ 单一配置源：所有子系统读取同一配置
- ✅ 外部挂载：修改配置无需重建镜像
- ✅ 版本无关：升级镜像时配置保持不变
- ✅ 易于备份：只需备份 config.yaml 和数据目录
- ✅ 多环境支持：通过不同配置文件切换环境

**配置模板管理**：
- `config.example.yaml`：提交到 Git（不含敏感信息）
- `config.yaml`：实际配置，不提交到 Git

**配置验证**：
- 启动时检查必要配置项（数据库路径、JWT 密钥等）
- 配置项缺失时启动失败并提示
- 生产环境自动禁用调试模式

### 6.4 环境变量

**使用场景**：

**管理方式**：
- `.env` 文件：本地存储，不提交到 Git

## 七、安全设计

### 7.1 认证与授权

| 机制 | 说明 |
|------|------|

| Agent 启用/禁用 | 操作者、时间、Agent ID |
