# 14-任务管理系统

## 一、概述

### 1.1 背景

1Claw 需要一套内建的任务管理系统，使人类用户能够通过 Web 客户端与 OpenClaw 进行任务层面的交互和管理。使用内建系统而非完全依赖第三方，是为了确保核心工作流程的可控性和一致性。

### 1.2 设计原则

- **内建优先**：核心任务管理功能由 1Claw 原生提供
- **API 驱动**：OpenClaw 完全通过 API 操作任务，无界面依赖
- **人机协作**：人类和 OpenClaw 都可以创建任务，但任务只能分配给 OpenClaw 执行
- **状态透明**：任务状态变更对所有相关方可见
- **通知及时**：状态变更和内容修改自动通知相关人员，超期自动提醒
- **扩展预留**：架构设计预留第三方任务系统对接能力

---

## 二、核心概念

### 2.1 任务（Task）

任务是 1Claw 系统中的基本工作单元，代表一项需要完成的工作或目标。

### 2.2 角色定义

| 角色 | 说明 |
|------|------|
| **创建人** | 发起任务的人或系统 |
| **执行者** | 被分配执行任务的 OpenClaw 实例 |
| **人类老板** | 具有最高权限的人类用户，可查看所有任务 |
| **验收人** | 负责确认任务完成质量的人（可与创建人相同） |

### 2.3 任务归属

- **创建者**：人类或 OpenClaw 都可以创建任务
- **执行者**：任务只能分配给 OpenClaw，不能分配给人类
- **验收者**：人类负责对 OpenClaw 完成的任务进行验收
- OpenClaw 可以创建子任务并自我分配，但父任务必须来自人类或上层任务

## 三、任务状态机

### 3.1 状态定义

| 状态 | 英文标识 | 说明 | 谁可设置 |
|------|----------|------|----------|
| **草稿** | `draft` | 任务正在编辑中，尚未正式提交 | 创建人 |
| **新建** | `new` | 任务已提交，等待分配或确认 | 创建人 |
| **待处理** | `pending` | 任务已分配，等待执行者开始 | 创建人/系统 |
| **处理中** | `in_progress` | 执行者正在执行任务 | 执行者 |
| **已完成** | `completed` | 执行者完成任务，等待验收 | 执行者 |
| **已验收** | `accepted` | 任务通过验收，正式关闭 | 验收人/创建人 |
| **已终止** | `terminated` | 任务被取消，不再执行 | 创建人/人类老板 |
| **已挂起** | `suspended` | 任务暂时暂停，后续可能恢复 | 创建人/执行者 |
| **已驳回** | `rejected` | 任务被拒绝执行，需重新评估 | 执行者 |

### 3.2 状态流转图

```
草稿 → 新建 → 待处理 → 处理中 → 已完成 → 已验收
              ↓         ↓         ↓
           已终止    已挂起    已驳回
              ↑         ↑
           (可恢复)  (可恢复)
```

### 3.3 状态变更规则

| 从状态 | 可到状态 | 条件 |
|--------|----------|------|
| 草稿 | 新建、已终止 | 创建人提交或删除 |
| 新建 | 待处理、已终止、已驳回 | 分配后或拒绝 |
| 待处理 | 处理中、已挂起、已终止 | 执行者开始或暂停 |
| 处理中 | 已完成、已挂起、已终止、已驳回 | 执行者完成或遇到问题 |
| 已完成 | 已验收、处理中 | 验收通过或需返工 |
| 已验收 | (终态) | 任务关闭 |
| 已终止 | (终态) | 任务关闭 |
| 已挂起 | 待处理、已终止 | 恢复执行或取消 |
| 已驳回 | 新建、已终止 | 重新评估后 |

## 四、功能需求

### 4.1 任务创建

**参与者**：人类、OpenClaw

**功能描述**：
- 创建人可以创建新任务
- 创建时需填写任务标题、描述、优先级、期望完成时间
- 任务初始状态为「草稿」
- 提交后状态变更为「新建」

**必填字段**：
- 任务标题
- 任务描述
- 截止时间（任务必须在此时间前完成）

**可选字段**：
- 优先级（低/中/高/紧急）
- 标签/分类
- 附件/参考资料

### 4.2 任务查看

**参与者**：人类、OpenClaw

**功能描述**：
- 人类老板可查看所有任务
- 创建人可查看自己创建的任务
- 执行者可查看分配给自己的任务
- 支持按状态、优先级、时间范围筛选
- 支持搜索（标题、描述、标签）

### 4.3 任务分配

**参与者**：人类、OpenClaw（系统）

**功能描述**：
- 创建人可将任务分配给指定 OpenClaw 实例
- 分配后任务状态从「新建」变更为「待处理」
- 系统自动通知执行者

### 4.4 任务执行

**参与者**：OpenClaw

**功能描述**：
- 执行者接收任务后，确认无问题则开始执行
- 开始执行时状态变更为「处理中」
- 执行过程中可更新任务进度/备注
- 执行完成后状态变更为「已完成」

### 4.5 任务验收

**参与者**：人类（创建人/验收人）

**功能描述**：
- 验收人查看已完成的任务
- 确认任务质量后，状态变更为「已验收」
- 如不满足要求，可驳回至「处理中」或「新建」

### 4.6 任务修改

**参与者**：创建人、执行者、人类老板

**功能描述**：
- 创建人可修改任务内容（标题、描述、优先级等）
- 执行者可修改任务备注/进度
- 修改后自动通知相关人员
- 修改历史记录需保留

### 4.7 任务取消/终止

**参与者**：创建人、人类老板

**功能描述**：
- 创建人可随时终止自己创建的任务
- 人类老板可终止任何任务
- 终止后状态变更为「已终止」
- 需填写终止原因

### 4.8 任务挂起/恢复

**参与者**：创建人、执行者

**功能描述**：
- 执行中遇到阻塞时可挂起任务
- 挂起后状态变更为「已挂起」
- 条件满足后可恢复至「待处理」或「处理中」
- 需填写挂起/恢复原因

### 4.9 任务评论

**参与者**：创建人、执行者、人类老板

**功能描述**：
- 任何有权限查看任务的人都可以添加评论
- 评论支持富文本，可@提及相关人员
- **状态变更时可填写评论**：改变任务状态时系统可提示填写变更原因或说明
- 评论按时间顺序展示，支持回复（线程式讨论）
- 评论记录永久保存，作为任务历史的一部分

**评论触发场景**：
- 任务分配时：执行者可回复确认或提出问题
- 状态变更时：填写变更原因（如驳回原因、挂起原因、终止原因等）
- 执行过程中：汇报进度、请求协助、说明问题
- 验收时：验收意见或改进建议

## 五、通知机制

### 5.1 通知触发场景

| 触发事件 | 通知对象 | 通知内容 |
|----------|----------|----------|
| 任务分配 | 执行者 | 新任务已分配，标题、描述、截止时间 |
| 任务确认接收 | 创建人 | 执行者已确认接收任务 |
| 任务开始执行 | 创建人、人类老板 | 任务状态变更为处理中 |
| 任务完成 | 创建人、人类老板 | 任务已完成，等待验收 |
| 任务验收通过 | 执行者、创建人 | 任务已验收关闭 |
| 任务驳回 | 执行者、创建人 | 任务被驳回，需重新处理 |
| 任务内容修改 | 执行者 | 任务内容被创建人修改 |
| 任务状态变更 | 创建人、执行者、人类老板 | 状态变更详情及原因 |
| 任务挂起 | 创建人、人类老板 | 任务挂起及原因 |
| 任务恢复 | 创建人、人类老板 | 任务恢复执行 |
| 任务终止 | 执行者、人类老板 | 任务终止及原因 |
| **截止时间提醒** | 创建人、执行者、人类老板 | 任务即将到期或已超期 |

### 5.2 通知方式

- Web 客户端站内通知
- 可选：邮件通知
- 可选：消息推送（如集成 Telegram/WhatsApp）

### 5.3 通知设置

- 用户可自定义通知偏好
- 可设置免打扰时段
- 可设置重要任务强提醒

### 5.4 截止时间提醒规则

| 提醒时机 | 通知对象 | 提醒内容 |
|----------|----------|----------|
| 到期前 24 小时 | 执行者、创建人 | 任务即将到期，请注意完成时间 |
| 到期前 2 小时 | 执行者、创建人 | 任务即将到期（紧急提醒） |
| 已超期未处理 | 执行者、创建人、人类老板 | 任务已超期，请立即处理 |
| 已超期已完成 | 创建人、人类老板 | 任务已完成但已超期，请验收 |

**提醒频率**：
- 超期后每天提醒一次，直到任务状态变更为「已完成」或「已终止」
- 已挂起的任务不触发超期提醒

## 七、API 设计原则（概念层）

### 7.1 API 使用方

- **OpenClaw**：通过 API 执行所有任务操作
- **Web 客户端**：通过 API 与后端交互
- **未来第三方系统**：通过 API 集成对接

### 7.2 API 资源

| 资源 | 操作 | 说明 |
|------|------|------|
| `/tasks` | GET, POST | 任务列表、创建任务 |
| `/tasks/:id` | GET, PUT, DELETE | 任务详情、更新、删除 |
| `/tasks/:id/status` | PUT | 变更任务状态 |
| `/tasks/:id/assign` | PUT | 分配任务 |
| `/tasks/:id/comments` | GET, POST | 任务评论 |
| `/tasks/:id/attachments` | GET, POST | 任务附件 |
| `/notifications` | GET | 通知列表 |
| `/notifications/:id/read` | PUT | 标记通知已读 |

### 7.3 API 认证

- 人类用户：Session/Cookie 或 JWT
- OpenClaw：API Key 或内部令牌
- 第三方系统：OAuth 2.0 或 API Key

## 八、数据模型（概念层）

### 8.1 任务（Task）

```
Task
├── id              任务唯一标识
├── title           任务标题
├── description     任务描述
├── status          任务状态
├── priority        优先级
├── creator_id      创建人 ID
├── assignee_id     执行者 ID
├── accepter_id     验收人 ID
├── created_at      创建时间
├── updated_at      更新时间
├── due_at          期望完成时间
├── completed_at    实际完成时间
├── accepted_at     验收时间
├── terminated_at   终止时间
├── suspended_at    挂起时间
├── tags            标签列表
└── metadata        扩展字段
```

### 8.2 任务历史（TaskHistory）

```
TaskHistory
├── id              记录 ID
├── task_id         关联任务 ID
├── action_type     操作类型（状态变更/内容修改/分配等）
├── old_value       旧值
├── new_value       新值
├── actor_id        操作人 ID
├── actor_type      操作人类型（human/agent）
├── reason          操作原因
└── created_at      操作时间
```

### 8.3 通知（Notification）

```
Notification
├── id              通知 ID
├── task_id         关联任务 ID
├── recipient_id    接收人 ID
├── type            通知类型
├── title           通知标题
├── content         通知内容
├── is_read         是否已读
├── created_at      创建时间
└── read_at         阅读时间
```

### 8.4 评论（Comment）

```
Comment
├── id              评论 ID
├── task_id         关联任务 ID
├── parent_id       父评论 ID（支持回复，为空表示根评论）
├── author_id       评论者 ID
├── author_type     评论者类型（human/agent）
├── content         评论内容（富文本）
├── is_system       是否为系统自动评论（如状态变更说明）
├── created_at      创建时间
└── updated_at      更新时间
```

## 九、权限模型

### 9.1 角色权限矩阵

| 操作 | 创建人 | 执行者 | 人类老板 | 其他 |
|------|--------|--------|----------|------|
| 查看自己创建的任务 | ✅ | - | ✅ | - |
| 查看分配给自己的任务 | - | ✅ | ✅ | - |
| 查看所有任务 | - | - | ✅ | - |
| 创建任务 | ✅ | ✅ | ✅ | - |
| 编辑任务内容 | ✅ | - | ✅ | - |
| 分配任务 | ✅ | - | ✅ | - |
| 开始执行 | - | ✅ | - | - |
| 完成任务 | - | ✅ | - | - |
| 验收任务 | ✅ | - | ✅ | - |
| 驳回任务 | ✅ | ✅ | ✅ | - |
| 挂起任务 | ✅ | ✅ | ✅ | - |
| 恢复任务 | ✅ | ✅ | ✅ | - |
| 终止任务 | ✅ | - | ✅ | - |
| 删除任务 | ✅ | - | ✅ | - |

### 9.2 权限说明

- **创建人**：对自己创建的任务有完整管理权限
- **执行者**：对分配给自己的任务有执行相关权限
- **人类老板**：拥有系统最高权限，可操作所有任务
- **其他用户**：默认无任务访问权限（可扩展协作场景）

## 十、第三方集成（未来扩展）

### 10.1 集成目标

允许外部任务管理系统与 1Claw 任务系统对接，实现：
- 双向任务同步
- 状态变更同步
- 通知互通

### 10.2 支持的系统（未来）

- Jira
- Trello
- Asana
- ClickUp
- 飞书任务
- 钉钉任务
- 企业微信任务

### 10.3 集成方式

- **Webhook**：接收外部系统事件

- 1Claw 为任务主系统，外部系统为镜像

- 冲突解决策略：时间戳优先 / 人工确认

## 十一、非功能性需求

### 11.1 性能

- 任务列表加载时间 < 2 秒
- 状态变更响应时间 < 1 秒
- 支持 10000+ 任务数据存储

### 11.2 可靠性

- 任务状态变更需保证原子性
- 通知发送失败需有重试机制
- 关键操作需有日志审计

### 11.3 安全性

- API 访问需认证授权
- 敏感操作需记录审计日志
- 防止越权访问（用户只能访问权限内任务）

### 11.4 可用性

- Web 客户端支持主流浏览器
- 支持响应式布局（桌面/平板/手机）
- 支持深色模式（可选）
