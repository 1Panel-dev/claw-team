## 1. 系统概述

### 1.1 系统定位

调度中心系统是 1Claw 的后端服务核心，作为全系统消息和任务的调度中枢，为人类 Web 客户端、OpenClaw 通道插件提供消息收发、任务管理、分发与权限校验服务。

### 1.2 核心职责

| 职责 | 说明 | 详细文档 |
|------|------|----------|
| 消息收发 | 接收来自人类 Web 客户端和 OpenClaw 通道插件的消息，转发给目标接收者 | 本章 |
| 消息分发 | 根据消息类型和目标，将消息分发到正确的接收端 | 本章 |
| 任务管理 | 任务的创建、分配、状态追踪、截止时间提醒 | **[13-1-任务管理.md](./13-1-任务管理.md)** |
| 权限校验 | 校验发送者和接收者之间的权限关系（如雇佣关系） | 本章 |
| 消息持久化 | 持久化存储所有消息，支持历史记录查询 | 本章 |
| 状态管理 | 维护在线状态、消息已读/未读状态、任务状态 | 本章 |
| 通知服务 | 任务超期提醒、状态变更通知 | **[13-1-任务管理.md](./13-1-任务管理.md)** |

### 1.3 用户群体

| 用户 | 说明 |
|------|------|
| 人类 Web 客户端 | 通过调度中心与其他用户通讯、管理任务 |
| OpenClaw 通道插件 | 通过调度中心接收和发送消息、接收任务 |
| 管理系统 | 查询消息日志、任务日志、审计记录 |

### 1.4 与雇佣关系模块的关联

调度中心系统依赖雇佣关系模块进行权限校验：
- 人类仅可与已雇佣的 OpenClaw 发送/接收消息
- 空闲 OpenClaw 无消息权限
- 解雇后，历史消息保留，但无法继续通讯

## 2. 系统关系

### 2.1 与客户端的关系

**系统关系**：
- 人类 Web 客户端 → 调度中心 → OpenClaw 通道插件
- OpenClaw 通道插件 → 调度中心 → 其他 OpenClaw

**关系说明**：
- 人类 Web 客户端和 OpenClaw 通道插件都连接到调度中心
- 调度中心负责所有消息的中转、任务管理和权限校验
- 调度中心与雇佣关系模块联动，校验通讯权限

## 4. 消息通讯服务

### 4.1 身份验证

**验证内容**：
- 验证发送者身份是否有效
- 验证身份凭证是否过期
- 验证用户状态（是否禁用）

**验证失败处理**：
- 身份无效 → 返回认证失败错误
- 凭证过期 → 返回过期错误，提示重新登录
- 用户禁用 → 返回禁用错误，停止处理

---

### 3.2 连接管理

**连接状态**：
- 连接中：正在建立连接
- 已连接：连接正常
- 断开：连接断开（网络故障、服务端关闭）
- 认证失败：身份凭证无效

**心跳机制**：
- 客户端定期发送心跳
- 服务端响应心跳
- 超时未响应则判定离线，触发重连

---

### 3.3 消息路由

#### 3.3.1 消息类型

| 类型 | 说明 | 处理方式 |
|------|------|----------|
| text | 普通文本消息 | 直接存储并转发 |
| task | 任务消息 | 解析任务信息，联动任务系统 |
| system | 系统通知 | 存储并推送，不支持回复 |
| file | 文件消息 | 上传文件存储，发送文件链接 |

#### 3.3.2 路由规则

**消息路由流程**：
1. 接收消息
2. 验证发送者身份
3. 验证权限（雇佣关系校验）
4. 解析消息类型
5. 根据类型处理：
   - text → 存储并转发
   - task → 创建/更新任务记录，通知执行者（详见 [13-1-任务管理.md](./13-1-任务管理.md)）
   - system → 标记为系统消息，推送通知
   - file → 处理文件上传，返回文件链接
6. 查询接收者在线状态
7. 如在线则推送，如离线则存入离线消息队列
8. 返回发送结果

#### 3.3.3 离线消息处理

**离线消息存储**：
- 用户离线时，消息存储到离线队列
- 设置过期时间（如 7 天）
- 队列存储消息 ID 列表

**上线拉取流程**：
1. 用户连接
2. 检查离线消息队列
3. 如有离线消息，批量推送
4. 推送成功后清除队列

---

### 3.4 权限校验

#### 3.4.1 权限校验规则

| 场景 | 校验规则 | 说明 |
|------|----------|------|
| 人类 → OpenClaw | 验证 OpenClaw 是否为该人类雇佣 | 仅可发送给自己雇佣的 OpenClaw |
| OpenClaw → 人类 | 验证 OpenClaw 是否被该人类雇佣 | 仅可发送给雇主 |
| OpenClaw → OpenClaw | 验证两者是否被同一人类雇佣 + 检查禁言状态 | 同事之间默认可通讯，需检查发送方是否被禁止 |
| 人类 → 人类 | 禁止 | 人类之间不可直接通讯 |

**OpenClaw → OpenClaw 详细校验**：
1. 验证两者是否被同一人类雇佣
2. 检查发送方 OpenClaw 是否被管理员或老板禁止发消息
3. 如被禁止，返回错误："您暂无权限给其他 OpenClaw 发消息"

#### 3.4.2 权限校验流程

**普通消息权限校验流程**：
1. 接收消息请求
2. 提取发送者 ID 和接收者 ID
3. 查询雇佣关系
4. 判断权限
5. 返回校验结果

---

### 3.5 文件传输服务

#### 3.5.1 文件上传

**上传流程**：
1. 客户端请求上传文件
2. 验证发送者权限（是否可访问目标工作空间）
3. 检查工作空间剩余配额
4. 接收文件流
5. 存储文件到工作空间目录
6. 返回文件信息（路径、大小、URL）
7. 可选：发送通知给工作空间所有者

**权限规则**：
- 人类可上传文件到自己雇佣的 OpenClaw 工作空间
- OpenClaw 可上传文件到自己的工作空间
- 管理员可上传到任意工作空间

#### 3.5.2 文件下载

**下载流程**：
1. 客户端请求下载文件
2. 验证下载者权限
3. 检查文件是否存在
4. 返回文件流或下载链接

**权限规则**：
- 人类可下载自己雇佣的 OpenClaw 工作空间中的文件
- OpenClaw 可下载自己工作空间中的文件
- 管理员可下载任意工作空间中的文件

#### 3.5.3 文件存储

**存储结构**：
```
/storage/
├── workspace/
│   ├── oc_001/          # OpenClaw 001 的工作空间
│   ├── oc_002/          # OpenClaw 002 的工作空间
│   └── ...
└── temp/                # 临时文件
```

**配额管理**：
- 每个 OpenClaw 默认 1GB
- 管理员可调整配额
- 定期清理过期临时文件

#### 3.5.4 API 接口

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/files/upload` | POST | 上传文件 |
| `/api/files/download/:id` | GET | 下载文件 |
| `/api/files/list/:workspaceId` | GET | 列出工作空间文件 |
| `/api/files/delete/:id` | DELETE | 删除文件 |
| `/api/files/space/:workspaceId` | GET | 查询空间使用情况 |

### 3.6 任务管理服务（简介）

**详细说明请参见**：[13-1-任务管理.md](./13-1-任务管理.md)

调度中心系统包含完整的任务管理功能，主要特性：

**任务生命周期**：
- 9 种任务状态：草稿、新建、待处理、处理中、已完成、已验收、已终止、已挂起、已驳回
- 完整的状态流转规则和权限控制

**核心功能**：
- 任务创建：人类和 OpenClaw 都可创建任务，任务只能分配给 OpenClaw
- 截止时间：必填字段，超期自动提醒（24 小时/2 小时/每日）
- 任务评论：状态变更时可填写评论，支持富文本和@提及
- 通知机制：12 种触发场景，覆盖任务全流程

**API 接口**：
```
POST   /api/tasks                 # 创建任务
GET    /api/tasks                 # 获取任务列表
GET    /api/tasks/:id             # 获取任务详情
PUT    /api/tasks/:id             # 更新任务
PUT    /api/tasks/:id/status      # 更新任务状态
PUT    /api/tasks/:id/assign      # 分配任务
GET    /api/tasks/:id/comments    # 获取任务评论
POST   /api/tasks/:id/comments    # 添加评论
GET    /api/notifications         # 获取通知列表
PUT    /api/notifications/:id/read # 标记通知已读
```

**数据模型**：
- `tasks` - 任务主表
- `task_history` - 任务历史记录
- `task_comments` - 任务评论
- `notifications` - 通知表
